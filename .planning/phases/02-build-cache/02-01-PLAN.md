The step body does 5 things in sequence within ONE pwsh step (vcvarsall env does not persist across steps):

**1. vcvarsall activation (CRITICAL -- no ilammy)**
Per RESEARCH.md Pattern 1 and CONTEXT.md locked decision:
```powershell
$installPath = '${{ steps.probe-msvc.outputs.install_path }}'
$selectedFull = '${{ steps.probe-msvc.outputs.selected_full_version }}'
$vcvarsall = Join-Path $installPath 'VC\Auxiliary\Build\vcvarsall.bat'

# Capture env before vcvarsall
$envBefore = @{}
Get-ChildItem env: | ForEach-Object { $envBefore[$_.Name] = $_.Value }

# Run vcvarsall in cmd subprocess, capture env after
$envLines = cmd /c "`"$vcvarsall`" x64 -vcvars_ver=$selectedFull >nul 2>&1 && set"
foreach ($line in $envLines) {
  if ($line -match '^([^=]+)=(.*)$') {
    $name = $Matches[1]; $value = $Matches[2]
    if (-not $envBefore.ContainsKey($name) -or $envBefore[$name] -ne $value) {
      Set-Item "env:$name" $value
    }
  }
}
$clPath = (Get-Command cl.exe -ErrorAction Stop).Source
Write-Host "vcvarsall activated: cl.exe at $clPath"
```
CRITICAL CONSTRAINT: Do NOT use ilammy/msvc-dev-cmd. Do NOT call vcvarsall in a separate step.

**2. CUDA paths**
```powershell
$env:CUDA_PATH = "$env:CONDA_PREFIX\Library"
$env:CUDA_HOME = "$env:CONDA_PREFIX\Library"
```
Use CONDA_PREFIX\Library, NOT $env:CUDA_PATH (per Phase 1 STATE.md).

**3. Version override via __init__.py rewrite (BLD-12)**
Per RESEARCH.md Pattern 3:
- Read base version from `llama_cpp/__init__.py` using regex `__version__\s*=\s*"([^"]+)"`
- Get llama SHA from `${{ steps.assert-submodule.outputs.llama_sha }}`
- IMPORTANT: assert-submodule outputs short SHA (currently 7 chars). For 12-char SHA per CONTEXT.md, re-derive with: `$llamaSha12 = (git -C vendor/llama.cpp rev-parse HEAD).Substring(0, 12)`
- Compute CUDA tag: extract major.minor from `${{ inputs.cuda_version }}`, strip dot, prefix with 'cu' (e.g. '12.6.3' -> 'cu126')
- Full version: `"$baseVersion+$cudaTag.ll$llamaSha12"` (e.g. `0.3.20+cu126.ll1a2b3c4d5e6f`)
- Rewrite `__init__.py` with the full version
- Log: `Write-Host "Wheel version: $fullVersion"`
- Emit output: `"wheel_version=$fullVersion" >> $env:GITHUB_OUTPUT`

**4. CMAKE_ARGS construction (BLD-03, BLD-04, BLD-08)**
Per RESEARCH.md Pattern 2 and CONTEXT.md locked decisions:
```powershell
$cudaArchs = '80-real;86-real;89-real;90-real;90-virtual'
$env:CMAKE_ARGS = @(
  '-G Ninja'
  '-DGGML_CUDA=on'
  "-DCMAKE_CUDA_ARCHITECTURES=$cudaArchs"
  '-DGGML_CUDA_FORCE_MMQ=ON'
  '-DGGML_AVX2=off'
  '-DGGML_FMA=off'
  '-DGGML_F16C=off'
  '-DCMAKE_C_COMPILER_LAUNCHER=sccache'
  '-DCMAKE_CXX_COMPILER_LAUNCHER=sccache'
  '-DCMAKE_CUDA_COMPILER_LAUNCHER=sccache'
  '-DCMAKE_POLICY_DEFAULT_CMP0141=NEW'
  '-DCMAKE_MSVC_DEBUG_INFORMATION_FORMAT=Embedded'
  '-DCMAKE_C_FLAGS_INIT="/Z7"'
  '-DCMAKE_CXX_FLAGS_INIT="/Z7"'
) -join ' '

$env:CMAKE_BUILD_PARALLEL_LEVEL = '2'
$env:SKBUILD_WHEEL_PY_API = ''   # Override pyproject.toml's py-api="py3" -> cpXX tag
```
Notes:
- Ninja generator per CONTEXT.md locked decision (BLD-08)
- sccache compiler launchers for C, CXX, CUDA (BLD-03)
- CMP0141=NEW + Embedded + /Z7 fallback via FLAGS_INIT (BLD-04, addresses sccache#2244 pitfall)
- CPU ISA disables per CONTEXT.md (AVX2, FMA, F16C off)
- CMAKE_BUILD_PARALLEL_LEVEL=2 (runner has 4 vCPUs but CUDA compilation is memory-heavy)
- SKBUILD_WHEEL_PY_API='' to get cp311-cp311 tag instead of py3-none (RESEARCH.md Pitfall 1)

Log CMAKE_ARGS for debugging: `Write-Host "CMAKE_ARGS: $env:CMAKE_ARGS"`

Document the fallback in a YAML comment above the SKBUILD_WHEEL_PY_API line.
